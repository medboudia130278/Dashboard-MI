How Subsystem Summary works

Export structure

Subsystem Summary builds virtual Excel files from the current project configuration. The Global project export creates one workbook for the selected project with all phases included. The Phase exports create one workbook per phase using the same sheet structure, filtered to that phase.

Calculated vs Imported override

  • Calculated means the rows are generated live from the current project setup: Project Phases, Cost Centers, Guide Planning, PIO/Freight/Customs, WBS, Support Costs, Price Lists, Currency & Exchange Rates, and Firming Rules.
  • Import Excel lets you replace the currently selected Subsystem Summary file with manually edited Excel data. The imported workbook must keep the same headers as the exported Subsystem Summary workbook so the preview, re-export and Mercury Interface can read it correctly.
  • If the selected file is the Global project export, the import replaces the full project package. If the selected file is a Phase export, the import replaces only that phase. Other phase exports remain calculated unless they are imported separately.
  • When an override is active, an Imported badge is shown. Upstream configuration changes are not reflected in that overridden file until you reset it.
  • Reset to calculated removes the imported override and rebuilds the rows from the current configuration.
  • Mercury Interface reads the active Subsystem Summary data exactly as displayed here. If a file is imported, Mercury uses the imported rows; if it is reset, Mercury uses the calculated rows again.
  • Imported Subsystem Summary overrides are included in the Cost Summary & MI JSON export/import. Excel source workbooks are still not included and must be imported separately through Data Sources.

Sheets

Each workbook contains one sheet per subsystem, named {Subsystem}_Synthesis. The project subsystems 3rd_Rail and CAT are grouped into a shared Feeding_System_Synthesis sheet. A dedicated Infra_Management_Synthesis sheet is also created for management workload, project-level materials, and project-level subcontracting rows.

Common output fields

The export headers are fixed so future types can reuse the same format. The generated rows fill Phase, Period, Type, Description, Quantity, Currency, Planning Guide, Costs Type, PBS/IBS, ABS, Associated WP, Tasks, Firming rule, Price List codes, and other pricing columns when a rule exists.

Text 1 to Text 9 columns

Each export sheet ends with Text 1 to Text 9. Their values come from the selected project's Price Lists Workspace text mapping. Project_name writes the project name, Phase writes the row phase, Phase code and Duration are looked up from Phase Timeline, and Not applicable leaves the cell empty.

When a Text column is mapped to Client Price List 1..9, the web preview keeps the cells empty, but the exported Excel file creates a real dropdown list in those cells. The dropdown values come from the manual Client Price List values configured for the same project.

Guide Planning export

The Export Guide Planning button creates a separate workbook named {ProjectName}_Guide_Planning_Export.xlsx for the selected project. It is complementary to the standard Subsystem Summary export and does not change the generated {Subsystem}_Synthesis sheets.

Sheet Purpose Main rules
Ovh_Renew_Planning Shows the yearly Overhaul and Renewal costs by Type, Phase, Subsystem, Currency and Planning Guide. Rows are included only when they already exist in the generated subsystem synthesis sheets. Year columns come from Year of Planning.
Planning_Guide_Profile Calculates the yearly percentage profile for each Planning Guide. Costs are grouped by Phase + Subsystem + Planning Guide. Multiple renewal lines are summed together, and currencies are converted to the Currency & Exchange Rates target currency before percentages are calculated.
Guide_Planning_Timeline Lists all non Overhaul/Renewal Planning Guides used by the synthesis sheets. Ovh_Workload and all Ovh/Renew types are excluded. Start date and End date come from Guide Planning Definition; Months and Years are calculated from those dates.

Workload rows

  • 1
    Subsystem workload rows use the generated WBS Workload table to know which Position, Subsystem and Period must be exported.
  • 2
    Supervisor, Technician and Worker rows are matched with the Cost Center Table. If a position exists for both Day and Night, both shifts are exported.
  • 3
    Quantity comes from Workload Synthesis quantities multiplied by the relevant Optimization coefficient rate for the phase, period and role.
  • 4
    Management workload rows go to Infra_Management_Synthesis. They use the remaining Cost Center positions, White Collar quantities, and the Manager or Engineer optimization rates depending on the position name.

Materials rows

  • 1
    Materials rows start from the Guide Planning material tables for MOB, REC and DEM. The Material Type becomes the export Description and the guide code becomes Planning Guide.
  • 2
    For subsystem sheets, Preventive spares, Corrective spares and Repair use imported Synthesis data. Currency comes from Synthesis when available, and External Purchase uses Yearly Total Cost or Yearly Reparable Cost.
  • 3
    For subsystem sheets, Tools, Consumables, PPE and Vehicles use their dedicated support-cost workspaces. Stored values are treated as EUR values and converted to the Project Currency through Currency & Exchange Rates. Price List Code 2 is Tools and Equipment for Tools, PPE and Vehicles; other Materials descriptions use Spares and Material.
  • 4
    Infra_Management_Synthesis receives Materials rows only for Other Support Costs, Tools, Consumables, PPE and Vehicles. Preventive spares, Corrective spares and Repair are ignored for this sheet.
  • 5
    For Infra Management, Other Support Costs uses the Other Support Costs table, Tools, Consumables and PPE aggregate Shared / Depot Pool plus MANAGEMENT, and Vehicles uses Project / Management.
  • 6
    Quantity is 1 when External Purchase is greater than 0; otherwise the row is ignored. MOB occurrence is 1, REC occurrence is the inclusive number of years, and DEM occurrence comes from Demobilisation Material Types months.
  • 7
    Costs Type, PBS/IBS, ABS, Associated WP and Tasks come from the WBS Materials table. For Infra Management, project-level WBS Materials rows without Subsystem are used.
  • 8
    Carat Unit and Unit Role come from PIO Definition. Firming rule comes from the Firming Rules table. Freight uses Onshore or Offshore Freight % depending on whether row Currency equals Project Currency. Infra Management keeps Insurances-Rates & Taxes empty.

Subcontracting rows

  • 1
    Subcontracting rows start from Guide Planning subcontracting tables for MOB, REC and DEM. Training is exported as Technical_Training; Obsolescence is exported as Obsolescence_Monitoring.

    The export uses Synthesis rows matching the sheet subsystem, Type and Currency. External Services Variable Cost comes from Yearly Cost (Subcontracting); DEM uses the same value divided by 12.
  • 3
    If several currencies exist for the same Description and subsystem, one row is generated per currency. Quantity is 1 when External Services Variable Cost is greater than 0; otherwise the row is ignored.
  • 4
    Costs Type, PBS/IBS, ABS, Associated WP and Tasks come from the WBS Subcontracting table. Carat Unit comes from PIO Definition, Firming rule comes from Firming Rules, and Freight plus Insurances-Rates & Taxes stay empty.

    Legal_Training is project-level and is exported only in Infra_Management_Synthesis. It does not use a subsystem and it is generated only when the Guide Planning subcontracting tables contain Legal_Training for the related phase and period.
  • 6
    For Infra Management Legal_Training, External Services Variable Cost comes from Mandatory Training Yearly total by phase. Values are treated as EUR and converted to Project Currency through Currency & Exchange Rates. DEM uses Yearly total / 12.

Overhaul & Renewals rows

  • 1
    Overhaul & Renewals rows start from the Overhaul & Renewals Guide Planning Table. Period stays empty and Nb Tot. of Occurency is always 1.
  • 2
    Overhaul creates Ovh_Workload and Ovh_Material. Renewal creates Renew_Management, Renew_T&C and Renew_Material.
  • 3
    The Total phase uses imported Synthesis totals filtered by Subsystem, Type and Currency. Other phases use yearly planning rows filtered by Phase planning years, Subsystem, Type and Currency.
  • 4
    Planning years are assigned to the phase containing 1 July of that year. The final project phase also includes its end year. If a phase starts exactly on 1 January, the next year after its end date is also included, but only when that year exists in the imported Overhaul/Renewal planning source.

How Mercury Interface works

Export structure

Mercury Interface creates one Excel workbook per selected virtual file. The Global project export contains one sheet named Infra_MI with all generated phases. Each Phase export keeps the same sheet and headers but filters generated child rows to the selected phase.

Infra_MI headers

The sheet uses the Mercury M_I interface headers, with Phase, Period, Type, Short Description and Total Cost added before Costing Tree. The export keeps one single sheet rather than one sheet per subsystem.

Source rows

The current implemented scope reads rows from Infra_Management_Synthesis, which is generated by Subsystem Summary. The Mercury mapping then places those rows under the relevant Costing Tree parents.

Subsystem parent blocks

After the Infra parents, Mercury Interface creates subsystem parent blocks only when the matching {Subsystem}_Synthesis sheet exists in Subsystem Summary. The order starts with Track, Feeding_System, POS, PSD, DEQ, VMI, then any remaining subsystem sheets.

The first subsystem block starts at 1.4, the next at 1.5, then 1.6, etc. Each block creates the same parent arborescence: Preventive, Corrective, Tools & Depot Investment, Common Facilities, Obsolescence, Technical Support, Technical Training, Overhaul/Renewals/Repairs, Accident and Vandalism.

Subsystem child rows

For each {Subsystem}_Synthesis sheet, child rows are inserted under the matching subsystem parent by checking the source row Type and Description. Workload preventive/corrective rows go under Preventive or Corrective Workload; Preventive spares and Corrective spares go under their Materials branches; Preventive_Subcontract and Corrective_Subcontract go under their Subcontracting branches.

Tools and Consumables go under Consumables, Standard & Special tools; Vehicles goes under Cars Fleet; PPE goes under Personal Protective Equipment; Obsolescence rows go under Obsolescence Monitoring; Technical_Support goes under Supplier Support Contract; Training and Technical_Training go under Technical Training Delivery.

Repair rows go under Repairs, Overhaul rows under Overhaul, and Renewal rows under Renewals. Matching accepts case differences, spaces and underscores, with the specific singular/plural variants currently defined for consumables, vehicles, repairs, renewals and obsolescence.

Costing Tree logic

  • 1
    Parent rows are created first from the predefined infrastructure hierarchy, for example 1 INFRASTRUCTURE, 1.1 Infra Project Management, and the lower management categories.
  • 2
    Parent rows keep Phase, Period and Type empty. They are structural rows used to organize the export.
  • 3
    Child rows are inserted under a parent when the source row Type and Description match the implemented rule for that Costing Tree branch.
  • 4
    Child Costing Tree values are generated by appending a sequential child number to the parent tree, such as 1.2.3.1.1, 1.2.3.1.2, and so on.

Mapped values

Most columns are copied by matching the Mercury header with the source Subsystem Summary header. Description from Subsystem Summary is copied to Mercury Short Description, while Long_Description is copied to Mercury Description. Other aliases include Carat Unit to CARAT Unit, Insurances-Rates & Taxes to Insurances, Rates, & Taxes, and External Purchase - Variable.

Total Cost

Workload: Nb Tot. of Occurency x Quantity x Cat 1 (Hours or Months) x Hourly rate. The Hourly rate is read from Cost Centers by matching the row Cost Centre.
Materials, Ovh_Material, Renew_Management, Renew_T&C and Renew_Material: Nb Tot. of Occurency x Quantity x (External Purchase - Variable + Freight per Unit + Insurances, Rates, & Taxes).
Subcontracting: Nb Tot. of Occurency x Quantity x (External Services Variable Cost + Freight per Unit + Insurances, Rates, & Taxes).
RISK: Nb Tot. of Occurency x Quantity x Fixed Contingencies/ Risk: Most likely value.
Ovh_Workload, parent rows and unknown types stay blank for now.

Implemented Infra rules

  • W
    Workload rows are distributed by exact or controlled description matches, for example Engineering rows under Engineering Management Workload, Planning rows under Planning Management, and Wayside Maintenance Manager under Routine Maintenance Management Workload.
  • M
    Materials rows are currently mapped for Tools, Vehicles, PPE and Other Support Costs under the related infrastructure parents.
  • S
    Subcontracting currently maps Legal_Training to Training Delivery.
  • O
    Overhaul, Renewals and Repairs Workload receives only Workload descriptions containing Overhaul, Renewal or Repair. Generic maintenance manager rows are excluded from this branch.

Risk Register rows

Risk rows come from the selected project's Risk Register Workspace. Imported rows are read from the Excel sheet Mercury_Config, then stored per project and reused by Mercury Interface.

They are inserted under 1.1.1.3 Project Risk, Savings and Opportunity Management. RISK ID is written to both Mercury Description and Short Description, Period stays empty, PBS maps to PBS/IBS, ASSOCIATED WP maps to Associated WP, and TASKS maps to Tasks. Both Mercury Currency columns receive the Risk Register Currency value.

The Global project export includes risk rows from all phases. Each Phase export includes only the risk rows whose Phase matches the selected phase.

Text 1 to Text 9 columns

The Mercury export reuses the same Price Lists Workspace configuration as Subsystem Summary. Project_name, Phase, Phase code, Duration and Not applicable are applied through the generated source rows, including Risk Register rows.

When a Text column is mapped to Client Price List 1..9, the web preview stays empty for that Text column, but the exported Infra_MI Excel sheet contains a real dropdown list in the corresponding cells.

Excel export behavior

The Excel export freezes the header row, applies an autofilter, and sets outline levels from the Costing Tree depth so the structure can be collapsed in Excel. The web preview also has vertical and horizontal scrollbars with sticky headers for easier review.

Current scope

At this stage Mercury Interface fills the Infra_MI sheet with the Infra Management mapped rows, creates the subsystem parent blocks, maps subsystem child rows from each available {Subsystem}_Synthesis sheet, and adds Risk Register rows under the Project Risk branch.

How WBS Workspace works

Overall logic

WBS generates editable mapping rows per project from the imported Excel sheets and the current project configuration. The main import reads Workload, Materials, Subcontracting, and Ovh & Renew; each card also has its own separate import button. Generated rows keep the Excel values for Costs Type, PBS/IBS, ABS, Associated WP, and Tasks.

Shared rules

  • P
    Rows are generated for each project phase and for the applicable periods MOB, REC, and DEM. If the Excel row has a Period value, only the matching period is tested.
  • M
    MOB is generated only when the phase has both Post Warranty Start and Post Warranty End.
  • S
    Subsystem names are matched against the project subsystems. FEEDING SYSTEM is expanded to 3rd_Rail and CAT. If no subsystem matches, the row is ignored unless it is an allowed project-level row.
  • D
    Duplicates are removed using Phase + Description/Position + Subsystem + Period + Costs Type + PBS/IBS + ABS + Associated WP + Tasks.

Workload table

Reads the Excel Workload sheet. White-collar rows use the Description text and the positions selected in Cost Centers. Technician and Supervisor rows use Workload Table quantities and Optimization coefficient rates. Worker rows use the Worker rate. Rows are generated only when the corresponding rate is greater than 0.

Materials table

Reads the Excel Materials sheet. Regular material rows depend on Guide Planning Material selections for MOB/REC/DEM. Project-level rows are allowed for Other Support Costs, Tools, PPE, and Vehicles. Other Support Costs is now also a Guide Planning material type for MOB, REC and DEM.

Subcontracting table

Reads the Excel Subcontracting sheet. Regular rows depend on Guide Planning Applicable Subcontracting Types. Technical_Training maps to Training, Obsolescence_Monitoring and Obsolescence_Treatment map to Obsolescence. Legal_Training stays project-level, with an empty Subsystem, and now follows the same MOB/REC/DEM Guide Planning rules as the other subcontracting types.

Overhaul & Renewals table

Reads the Excel Ovh & Renew sheet. Repair rows are REC only and require Synthesis Type containing Corrective plus Yearly Reparable Cost > 0. Overhaul and Renewal rows have an empty Period and require Synthesis Type matching Overhaul/Renewal plus Total Global Cost > 0. Rows without Subsystem are ignored.

Project-level rows

Description Rule
Other Support Costs MOB if Capex Cost > 0, REC if Annual Opex > 0, DEM if Demobilisation Material Types value is > 0 and Capex Cost is available for the same phase.
Tools Generated when Shared / Depot Pool has Individual, Collective, or Special tools > 0 for the same phase and period.
PPE Generated when MANAGEMENT has PPE > 0 for the same phase and period.
Vehicles MOB uses Capex, REC uses Fuel or Operating annual cost, DEM uses Capex, Fuel, or Operating cost from Project / Management.

Editable output

The WBS rows are generated from source rules, but the WBS classification fields remain editable in the table. Edits are stored per generated row key and reused when the workspace is rendered again.

How Mandatory Training works

Overall logic

The table estimates yearly mandatory training cost for each project phase. It uses the training catalogue rows, the phase list, the Workload Synthesis headcount, and the Headcount & Optimization rates. The generated table is Recurrent period only; Mobilization and Demobilization are not used in this workspace.

Workload calculation

  • 1
    Personnel Concerned is split by comma or semicolon. Each entry is matched against project subsystems, or against white-collar groups such as Engineers and Management.
  • 2
    For subsystem entries, workload is based on Workload Synthesis: Total Techs and Total Supervisors. The Recurrent rates from Headcount & Optimization are applied to Technician, Worker and Supervisor pools.
  • 3
    For Engineers and Management, quantities come from the White Collar table for the current phase and Recurrent period. If no override exists, the default quantity is 1 per selected position.
  • ALL
    If Personnel Concerned is ALL, the workload is the sum of all subsystem headcounts plus Engineers plus Management.

Cost formula

Case Formula
No group cost Yearly cost = Workload × Cost/Person ÷ Periodicity
Group cost available Cycle cost = full groups × Cost/Group + min(remaining people × Cost/Person, Cost/Group)
Final yearly cost Yearly cost = Cycle cost ÷ Periodicity

Full groups are calculated with floor(Workload / Max/Group). Remaining people are priced at Cost/Person, capped by Cost/Group so a partial group never costs more than a full group.

Automatic update

Changing the training catalogue, Headcount & Optimization rates, White Collar quantities, or Workload Synthesis data recalculates workload, sessions per cycle and yearly cost when the workspace is rendered.

Currency display

The display currency selector applies the workspace exchange rate to the calculated yearly cost. The row Currency field remains editable and is kept for source traceability.

How Cost Centers works

Purpose

The Cost Centers workspace defines the project currency, annual working hours, applicable positions, time periods, cost centers and hourly rates. These values are reused by WBS, Subsystem Summary, Mercury Interface and several export calculations.

Project-level setup

  • Project Currency is the reference currency used to decide Onshore versus Offshore and to populate export currency defaults.
  • Annual Working Hours comes from General Parameters when available, otherwise from manual override.
  • Night uplift % can be applied to night rows when night hourly rates need an uplift from day rates.

Generated table rows

  • Rows are generated from the selected Applicable Positions.
  • Supervisor, Technician and Worker use the general Day/Night time-period selector.
  • Engineer and other white-collar positions use the engineer time-period selector.

Column Role

Time Period Usually Day or Night. If a position exists in both periods, exports create separate rows for each period.

How Project Phases works

Purpose

The Project Phases workspace defines the timeline of each project. These dates, phase names and phase codes are reused by Guide Planning, Headcount, WBS, Subsystem Summary, Mercury Interface, Price Lists text mapping and phase exports.

Phase identity

  • Phase is the business phase name used in workspaces, WBS rows and export filters.
  • Phase code is reused by Price Lists text mapping and export sheets.
  • Duration is used as a phase duration reference, especially in Text mapping when Duration is selected.

Phase dates

  • Start date and End date define the phase window used by phase exports and planning year allocation.
  • Post Warranty Start and Post Warranty End mark the post-warranty window used by MOB logic in several downstream modules.

Main downstream rules

Area How Project Phases is used
Guide Planning Definition Generates MOB, REC and DEM guide planning rows per phase. Recurrent tables use phase dates to calculate durations in months or years.
Headcount & Optimization Creates coefficient and white collar rows by Phase x Period. Each phase receives the applicable period rows.
WBS Workspace Generates Workload, Materials, Subcontracting and Overhaul & Renewals rows per phase. MOB rows depend on the phase warranty/post-warranty setup.
Subsystem Summary Builds global and phase Excel exports. Phase exports only keep rows matching the selected phase.
Mercury Interface Builds global and phase Infra_MI exports using the same phase filtering logic as Subsystem Summary.
Price Lists Text mapping When Text mapping uses Phase, Phase code or Duration, the values are read from the configured phase timeline.

MOB and post-warranty behavior

For modules that need a mobilization period, a phase generally needs both Post Warranty Start and Post Warranty End. If they are missing, MOB rows can be skipped or generated with default zero values depending on the target module.

Overhaul and renewal planning years

Planning years are assigned to phases using the current year-window rule: a year belongs to a phase when July 1 of that year falls inside the phase. The final project phase also includes its end year. A specific January 1 boundary handling avoids losing the last source planning year while preventing overlap with the next phase.

Recommended workflow

  • Configure Project Phases before Guide Planning, WBS, Subsystem Summary and Mercury Interface.
  • After changing phase dates, phase codes or warranty dates, refresh and review downstream workspaces because many generated rows depend on this timeline.
  • Keep phase names stable when possible, because imported JSON configuration and generated WBS rows are linked to phase names.

How Workload Synthesis works

Purpose

The Workload Table converts imported Synthesis and Hours Report workbook data into one row per Subsystem x Shift. These rows are reused later by Headcount & Optimization, WBS, Subsystem Summary, and Mercury Interface.

Hours Report source

For each Subsystem and Shift Type, the code sums hours_worked into Preventive Hrs and sums available_days into available days. Shift names are normalized to Day or Night.

Synthesis source

For each Subsystem and shift, Synthesis provides paliative_hours_corrective, day_technicians_optimized and night_technicians_optimized. Corrective Synthesis rows also provide yearly_total_hours_corrective per Subsystem.

Generated columns and formulas

Column Formula / rule
Subsystem Unique subsystem list from Synthesis, in first appearance order. Blank subsystems are ignored.
Shift Two generated rows per subsystem: Day and Night.
Preventive Hrs Sum of Hours Report hours_worked for the same Subsystem and Shift.
Day/Night % Preventive Hrs for this row divided by total Preventive Hrs of the subsystem across Day + Night. If total is 0, the rate is 0.
Remaining Hrs Hours Report available_days x 8 for the same Subsystem and Shift.
Paliative Hrs Synthesis paliative_hours_corrective for the same Subsystem and Shift. The first matching value is used.
Corrective Hrs Synthesis yearly_total_hours_corrective for the subsystem x Day/Night %.
All Corrective Hrs Paliative Hrs + Corrective Hrs.
Corr/Prev % All Corrective Hrs divided by Preventive Hrs. If Preventive Hrs is 0, the rate is 0.
Gap Hrs Remaining Hrs - All Corrective Hrs. Negative values are highlighted and counted as gap alerts.
Prev. Techs If Preventive Hrs + All Corrective Hrs > 0: Preventive Hrs / (Preventive Hrs + All Corrective Hrs) x optimized technicians for the shift.
Corr. Techs Optimized technicians for the shift - Prev. Techs.
Prev. Supervisors Supervisor pool is 1 when total technicians for the subsystem are <= 8, otherwise 2. Prev. Supervisors = Prev. Techs / total technicians x supervisor pool.
Corr. Supervisors Corr. Techs / total technicians x supervisor pool.

Rounding and alerts

Most generated numeric values are rounded to two decimals. Percent columns are stored as ratios internally and displayed as percentages. A negative Gap Hrs means corrective workload exceeds remaining capacity for that Subsystem and Shift.

Manual actions

  • Import Workload Excel replaces the auto-computed rows for that project with rows from an Excel sheet whose name contains Workload. The import expects Subsystem and Shift columns, plus the displayed workload columns.
  • Reset to auto removes imported rows and returns to the calculation from Synthesis and Hours Report.
  • Edit lets you override individual computed cells. Those overrides are saved per project.
  • Add row creates a manual row for additional Subsystem/Shift staffing values. Manual added rows only carry technician and supervisor counts; calculated hour columns stay blank.

Downstream usage

The resulting Prev./Corr. Technician and Supervisor values are used by Headcount & Optimization, WBS Workload, Subsystem Summary, and the Mercury Interface. Imported rows and manual rows are included in those downstream calculations.

How Headcount & Optimization works

Overall logic

This workspace bridges two upstream tables to produce the Headcount Summary: the Workload Table (from Workload Synthesis) provides Technician and Supervisor counts per subsystem, while the White Collar Table (defined here) provides Engineer and Manager quantities per phase. The Optimization coefficient table contains percentage multipliers that control how many of those people are actually counted in each period.

Optimization coefficient table — column by column

  • T
    Technician rate % — share of the Workload Table technicians that remain classified as Technicians in the Headcount Summary. Example: 80 % keeps 8 out of 10 techs as Technicians.
  • W
    Worker rate % — share of the same Technician pool that is reclassified as Workers. Setting Technician = 80 % and Worker = 20 % splits 10 techs into 8 Technicians + 2 Workers. Default is 0 % (no Workers unless explicitly set).
  • S
    Supervisor rate % — share of the Workload Table supervisors that appear in the Headcount Summary. Use this to reduce the supervisor count below the automatic calculation. Example: 50 % halves the supervisor headcount for a given period.
  • E
    Engineer rate % — applied to the Engineer quantities defined in the White Collar Table. Not subsystem-dependent; Engineers are a project-level resource and appear only in the Project total rows of Headcount Summary.
  • M
    Manager rate % — same principle as Engineer rate, applied to Manager positions from the White Collar Table.

Period defaults

Period Technician Worker Supervisor Engineer / Manager
Mobilisation 0 % 0 % 0 % 0 %
Recurrent 100 % 0 % 100 % 100 %
Demobilisation 0 % 0 % 0 % 0 %

During Mobilisation, the default assumption is that nobody is on-site yet. Setting a coefficient — for example 50 % — means half the workforce is already present at the start of the phase; the other half joins at the beginning of Recurrent.

Headcount Summary is updated automatically

Any change to the Optimization coefficient table, the White Collar Table, or the Workload Table immediately recalculates the Headcount Summary — no manual refresh needed.
